What's Pizzia ?

Pizzia is an API which allows you to easily make modules for your own Zia. By respecting the rules and principles of Pizzia when you make your modules, you are guaranteed to make something coherent, which provides a high level of abstraction and allows your module to handle only one task

This follows the rules of the divide and conquer pattern in order to make it easier for you to complete your HTTP web server. Dividing your web-server into several modules carrying out micro-tasks is essential. Find the remaining of these explanations on the quickstartpage that we set up for you.

Why should I vote for Pizzia during the election ?

In order fo you to use Pizzia, we have to be elected. There are many reasons to opt for Pizzia rather than other APIs :
  • Pizzia offers a high level of abstraction, allowing you to be extremly productive and fully respect the RFC2616
  • Pizzia is fully documented, you can find it on our repository or website with a lot of diagrams. Like mentionned above, you have at your disposal a quickstart page to get impregnated with the philosophy of the API. A fully documented guide which explains how to implement each of the provided interfaces. Examples of pre-made modules that implement our API and finally a Doxygen to find the documentation of each function provided by our API.
    In short, there's a lot of documentation for you to work with !
  • Pizzia was made and thought in order to let you follow several design patterns. By implementing our API, you will be able to realize your own web server in the way of a pipeline (managed by a mediator for exemple).
  • We offer full support via Telegram or Discord (but also IRL ! =D) to help you using the API. If you have issues, we will be happy to exchange with you.
  • Everyone can freely contribute our API via GitHub, so don't hesitate to become a Pizzia contributor. In order to achieve this, you only need to refer the lowest part of the FAQ.

Where to start ?

First of all we advice you to read the short quickstart in order yo understand the philosphy of Pizzia in less of 5 minutes.
Then, you can pass to the technical guide which will explain you step by step how to implement Pizzia. Once you implement the different interfaces provided by Pizzia, it will be time to make you first modules.
You can find examples made by the team in order to help you.
At any time, you can find a very usefull Doxygen of the API while you're working.

How to contribute Pizzia ?

It will be a pleasure to receive your contributions to Pizzia, but first of all you must know what type of contribution you want to make:
  • Minor improvement: For example: a spell check on the docs or the adding of informations on our website...
  • Major improvement: The change of a method's name or the adding of new methods...
The difference between these two types of improvements are defined by what they involve: a major improvement will involve to notify all the people using our API that a modification has been made and must be updated in their project. So this type of improvement really matters and it will only be approved if all requirements are respected:
  • The improvement is justified by a problem of coherence proved about the actual code of Pizzia. For example, if you think one of the methods doesn't respect the RFC2616.
  • The improvement is justified by a simple and strong explanation when submitting the pull-request
  • The imrovement doesn't have a very hudge impact and doesn't require a consequent change to make for all of the teams using a working Pizzia API
  • The improvement does not disagree with the philosopy of Pizzia API
  • The improvement may be modified by our team in the case where we want to integrate it in a different way in Pizzia
  • The improvement is approved by at least four members of the Pizzia team when submitting the pull-request

As you can see, a major improvement must respect some rules to be approved. All these rules have been setup to avoid all groups to continually update their code. We want that all groupe using Pizzia API work in serenity

All minor improvement can be easily justified by an explanation when submitting the pull-request and must be approved by at least two members of the team. In order to contribute, you must start by forking our repository. This will allow you to have a copy of our repository on your Github account in order to make the desired change without touching to our primary repository.
Once forked, you can clone it and make a new branch: git checkout -b the-modification-description

Name must be explicit and written in english by using the kebab-case otherwise the improvement will be automatically refused.

Once you made the change on the new branch and that it has been committed and pushed on GitHub, it is time to offer us your change. For that, go to your GitHub URL. You must have a view similar to this one:

You just have to click on Compare & pull request, you will be invited to explain your improvement in english. Please respect all the rules previously explained when submitting the pull-request.

Once your pull-request has been submitted we will analyze it in a delay of 48 hours.
In some case we will notify you that we need you to argue about your change. Finally according to all these rules, your pull-request shall be accepted or not by our team.
If you think your pull-request was wrongly refused, we invite you to get in touch with us on Telegram or Discord.

We invite you to follow this tutorial if you need more details on how to realize an improvement on GitHub.
