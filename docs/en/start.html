Introduction

Last Update: the 31st of January 2019

Pizzia, what is that ?

If you are here, it's probably because you are doing the Zia project by Epitech, that's good, the aim of Pizzia is to give you a simple and coherent way to build it by proposing you to respect and implement and API applying to your HTP modules.

Thanks to Pizzia, you will be able to build easily and quickly your ZIA architecture by following our explanations and diagrams as well as achieving your different modules in a clear and playful way.

Throughout your project, we are commited to answer any questions related to the API or related to your architecture design by following the API philosophy, besides we are also commited to help you (as far as possible) to implement Pizzia's API.

Finally, you have the possibility to contribute to Pizzia (we would be very pleased to take in count your changes) and to discuss with us about something that does not please you on our API (i.e. a design choice, a function name),we think we could continuously improving ourselves and thus we are always open to dialog. To do so, we invite you to visit our FAQ page to find out how to participate to our project.

Our Philosophy

Our API philosophy can hold in brief: simplicity. Our goal is to provide an API with a high abstraction level to ensure that your modules could be writen in a most concise way.

Similarly to what UNIX says, (write programs that do one thing and do it well) we want to allow you to build modules that perform one unique task in a way to dispatch the process complexity of a web server in different modules.

Pizzia takes in consideration and respect the RFC2616 from various perspectives (function names for example) in order to offer an universally understable API.

Our Architecture

In order to instill the Pizzia's API philosophy, we will (without going into detail) present you a web server architecture with its different modules implementing Pizzia.

First of all, start to think high level to get Pizzia's operation. We won't talk about socket, nor threads as examples, because Pizzia won't impose you any constraint on how building low level things on your web server.

We could sum up (in a synthetic way) an HTTP request made by your brower to a web server with the following diagram:

Indeed, we can see the client (your web browser) sending an HTTP request to the web server (Apache, NGINX or your Zia).
This request contains a bunch of informations (asked resource, method, headers, content ...) that will allow the server to understand the request to treat afterwards, then resend a response in return that also contains bunch of informations (response status, headers, content ...) allowing the web browser to display it to the final user.

So we quickly understand our web server job lies in the fact of receive a request then perform the n√©cessary treatment in order to build the response. Well, it's precisely what Pizzia want you to think on how to construct your modules. So let's continue on our diagram:

As you might have guessed, the different modules that were added on our diagram are in fact modules implementing Pizzia's API !

To make it simple, a module is an object implementing the Pizzia::IModule interface. That will receive an object containinf the request that implements Pizzia::IRequest interface as well as an object containing the response that implements the Pizzia::IResponse interface. From there, the module will properly work, it will be able to modify the request or the response in order to make the next module available etc.

This is called a module pipeline executing modules one by one (with a priority order) relieving the request and the response, modified or not.

Let's sum up the request execution flow with the pipeline execution with modules we showed earlier:

  • First of all, the web serveur will receive the request (some low level): it will receive a socket and read its content. In our case, our web server is a HTTPS server, the read datas on the socket are ciphered. It will build an object contaning an empty response (filled by the modules) and an object containing the raw and the ciphered request (that can be changed by the modules). It will then call the first pipeline module by sending the two objects.

  • The first module: SSL_decrypt will decipher the request content, so it will modify the object containing the request so that this one contains the decrypted request, however it will not change the response. This module only take action on the request, its goal is precise and simple: decipher it.

  • The second module: PHP will take care of executing the PHP scirpt associate to the asked resource. Through CGU, it will transmit the request (gracefully decipher by the last module) to PHP that will interpret the script, the response from the PHP interpreter will be used to modify the object containing the resoonse (so far empty)

  • The third module, compression module will simply be in charge of compressing the response content. It will first read the request headers in order to determine if the client is able to decompress the response.If so, it will modify the response (that was already modified before by the PHP module) in order to compress it.

  • The fourth and last module: SSL_encrypt will make the oposite action tothe first module (SSL_decrypt), it will simply decipher the response (just compressed by the last module). This one doesn't take care of request datas, it will only modify the response.

  • Last, the pipeline execution is done: we have now an object containing the reponse filled by the modules. All that's left is to write the response raw datas on a socket to transmit it to the web browser !

If you understood all of that, good news: you understood the Pizzia's philosophy and all of the underlying architecure.

We sumed up from a synthetic way the operation of our API, but the principle is really simple to assimilate: we just have chaining modules in order to manipulate the request/response.

Every module can modify, if needed, the request (so that the other modules executing afterwards can access to that modified request) as well as the reponse (for the last sending in particular, or for the following modules). When this chain of modules is done, the response is sent to the client (the last module has the hand on the response, as well as the first can completely modify the request).

Finally, this guide is synthetic and aiming to be light (we voluntarily ommited to write lines of code) to make you understand the operation and the philosophy of Pizzia. We wrote a technical guide with code and instructions to implement yourself Pizzia for your Zia, this guide very handsome aims to make you understand in depth the operation of Pizzia.

We provide you a Doxygen documentation of our API as well as some modules examples implementing Pizzia.

Where do i start ?

First of all, you need to download tha Pizzia API in order to begin to implement it. You have two solutions: - to directly use our repository on Github by going through the submodules system from GIT
- Direct download the last release of our API and then copy/pastein your project.

You have to know that during the project, the API es likely to be changed (after your feedbacks and after your contribution), that's why we advise you the first solution in order to have changes in real time.

Nevertheless you can use the second one. We are ommited to communicate on any changes or any new version (by e-mail / Yammer or anything) and also not making any major changes in order to ensure your that you won't have to remake everything. Only small changes, we promise.
